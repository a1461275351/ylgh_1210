### T08 + T09 交付要点(核心清关链路)

- **前置校验引擎** `server/src/ceb/validate.js`:6 道关卡 —— 商品备案、正面清单、账册余量、单次限值(5000)、个人年度额度(26000)、价格异常(预警不拦截)。FAIL 拦截、WARN 放行。
- **订单接入** `routes/orders.js`:单条 API / 批量导入 / 样例生成;接入即校验,状态机 已接收→已校验/校验失败→已生成清单→放行。支付单、运单登记与三单对碰监控。
- **清单生成与申报** `routes/inventories.js`:由已校验订单自动生成清单(拉账册号、按 HS 税率算综合税)→ 复用 T19 引擎申报 → 放行回写订单状态。退单工作台(批量重报)、撤销。
- **为支撑校验预建**:商品备案库、HS 税率库、电子账册、库存台账的表与种子数据(完整界面留待 T06/T07)。
- **前端**:三单数据中心(订单管理/支付单/运单/三单对碰监控)、清单管理(申报清单/退单工作台)。
- 累计测试:客户 15 + 合同 11 + 接口平台 16 + 三单清单 17 = **59 用例全通过**。

### T19 交付要点

- **报文引擎** `server/src/ceb/`:`xml.js`(XML 组装)、`messages.js`(CEB621 清单申报报文)、`tax.js`(综合税试算)、`sign.js`(加签插槽)、`channels.js`(通道适配 + 回执模拟器)、`engine.js`(编排)。
- **加签**:原型用 SHA-256 mock 占位;生产替换为电子口岸 IC 卡/证书加签(国密 SM2/SM3 或 RSA),`sign.js` 已留 `DIRECT_CARD` 分支。
- **通道双轨**:`SIMULATOR`(原型默认,按规则放行/退单)、`DIRECT`(单一窗口直连,已有电子口岸卡,待联调补 endpoint)、`THIRD`(第三方,备用)。配置可切换、互为灾备。
- **回执模拟器规则**:超 5000 元单次限值 / 缺订购人证件 / 商品不在正面清单(HS 以 999 示意)→ 退单(300);否则放行(120),回写清单编号。
- **报文留痕**:发送 CEB621 与接收 CEB622 全部落 `msg_log`(原文留存),支持查询、查看原文、失败重发。
- **前端**:清单管理→申报清单(生成样例→申报推送→详情/报文时间轴/XML 原文)、系统管理→通道配置、系统管理→报文日志。
- ⚠️ 待办:样例清单中订购人证件号为明文(演示),生产须 AES 加密存储、报文脱敏/密文传输。

```
① 备货进口入区:进境备货报关单(1210) → 放行 → 入区核注清单(核增账册+500)
                → 入区核放单 → 过卡入区

③ 出区发货:放行清单汇总 → 出区核注清单(核减账册-2) → 出区核放单
                → 过卡出区 → 订单置「已出库」
账册轨迹实测:期初 1500 → 备货入区 2000 → 出区发货 1998
```

## 每个功能任务的统一验收标准

1. 后端 API 按 REST 规范实现,参数校验齐全,错误返回 `{code, message}`;
2. 前端页面:查询条件区 + 结果表格 + 分页 + 操作(新增/编辑/删除/审核/申报),风格与参考截图一致(左侧深色菜单 + 顶部多页签);
3. 状态流转类单据(账册变更、清单、核注清单等)必须实现完整状态机,非法流转要拦截;
4. **测试通过才算完成**:
   - API 冒烟测试(curl 脚本或 test.http):正常路径 + 至少 2 个异常路径;
   - 浏览器验证:页面可打开、查询/新增/编辑闭环操作成功;
5. 涉及账册数量的操作必须通过 `ems_stock_ledger` 记流水,禁止直接改余额。

## 关键设计约定

- **多租户**:业务表带 `customer_id`(货主),自营主体 `is_self=1` 作为默认租户;
- **单据编号**:内部单号规则 `{类型前缀}{yyyyMMdd}{6位流水}`,如 `INV20260706000001`;
- **海关交互**:原型阶段全部走"回执模拟器"(接口平台内置),报文均落 `msg_log`,生产替换为真实通道适配器(DIRECT/THIRD 双轨);
- **个人信息**:身份证号 AES 加密存储,页面脱敏展示(`610***********1234`);
- **审计**:所有写操作记 `sys_op_log`。
