# 1210 关务系统 · 上线实施方案

> 面向从"功能版"到"生产上线"的两项实施工作:**① 切换 MySQL** 与 **② 对接真实单一窗口**。
> 两项均含代码之外的外部依赖,本方案给出完整落地步骤、责任方与前置条件。

---

## 一、切换生产数据库(SQLite → MySQL)

### 1.1 为什么要切
原型用内置 SQLite(单文件、零配置),适合演示与功能验收;生产要支撑并发、备份、大促单量,应换 MySQL 8。

### 1.2 前置条件(需业主提供)
- 一台 MySQL 8.0 实例(可本机、可云 RDS),及库账号密码;
- 网络可达(内网/专网部署时确保应用服务器能连到 MySQL)。

### 1.3 已就绪
- 生产建表脚本 `db/schema.mysql.sql`(43 张表,含注释、索引、初始参数/代码表种子);
- 表结构已与原型逐字段对齐(见验收报告)。

### 1.4 实施步骤
1. **建库导表**:在 MySQL 执行 `SOURCE db/schema.mysql.sql;`(或用客户端导入),得到全部表与初始参数;
2. **接入驱动**:后端引入 `mysql2`,新增连接池模块(读取环境变量 `DB_HOST/DB_PORT/DB_USER/DB_PASS/DB_NAME`);
3. **数据访问层适配**:原型 `server/src/db.js` 使用 node:sqlite 的**同步** API(`db.prepare(sql).get/all/run`);MySQL 驱动为**异步**,需将数据访问改为异步(`await`)。这是本项主要工作量,建议:
   - 封装统一 DAO:`query(sql, params)` / `get(sql, params)` / `run(sql, params)`;
   - 将各路由处理函数改为 `async`,`db.xxx()` 前加 `await`;
   - SQL 占位符 `?` 两者通用;少量方言替换:`datetime('now','localtime')`→`NOW()`,`ON CONFLICT ... DO UPDATE`→`ON DUPLICATE KEY UPDATE`,`INSERT OR IGNORE`→`INSERT IGNORE`(均已在代码注释标注);
4. **回归测试**:现有 167 用例的测试脚本无需改动(走 HTTP),连上 MySQL 后重跑 `npm test` 应全绿;
5. **配置与备份**:生产开启 MySQL 定时备份、慢查询日志;应用侧用环境变量管理连接信息,不写死。

### 1.5 工作量与验收
- 工作量:数据访问层异步化改造(集中在 `db.js` + 约 20 个路由文件的 await 化);
- 验收标准:连接生产 MySQL 后 `npm test` 167 用例全通过 + 手工全流程通过。

> 注:该改造需真实 MySQL 环境方可执行与验证。如业主提供 MySQL 实例(或授权在部署机安装),即可实施并现场验收。

## 二、对接真实单一窗口通道

### 2.1 现状(已就绪的部分)
接口平台已完成并可运行:
- **报文引擎**:CEB621 清单申报报文组装(`server/src/ceb/messages.js`);
- **加签插槽**:`server/src/ceb/sign.js` 预留 `DIRECT_CARD` 分支;
- **通道适配层**:`server/src/ceb/channels.js`,含 `SIMULATOR`(模拟器,现用)/`DIRECT`(直连,占位)/`THIRD`(第三方);
- **报文留痕**:收发报文全部落 `msg_log`,支持查看原文、失败重发。

即:业务侧"生成清单→组装报文→送出→接回执→回写状态"整条链路已跑通,当前对接的是**模拟器**。

### 2.2 前置条件(需业主/海关侧提供,代码无法替代)
1. **跨境电商企业备案**:电商企业/平台/支付/物流在海关完成备案(取得海关代码);
2. **电子口岸 IC 卡/Ukey + 加签中间件**:法人卡/操作员卡,及读卡签名服务(国密 SM2/SM3 或 RSA);
3. **单一窗口 / 通关服务平台接入信息**:企业账号、报文交换通道(QP 客户端 / 前置机 / 省级电子口岸网关)、接口地址与凭证;
4. **海关测试环境**:先在海关提供的**测试环境**联调,通过后再切生产;
5. 三单分工确认:订单(本系统推)、支付单(支付企业推)、运单(物流企业推)。

### 2.3 实施步骤
1. 办理/收集 2.2 的资质与接入信息;
2. **实现 DIRECT 通道适配器**:在 `channels.js` 的 `DIRECT` 分支填入真实报文交换逻辑(HTTP/文件/前置机),`endpoint` 配置到"通道配置"页;
3. **实现真实加签**:在 `sign.js` 的 `DIRECT_CARD` 分支接入电子口岸读卡签名中间件;
4. **报文规范核对**:以海关最新统一版报文规范(XSD)校准 CEB311/CEB621/CEB622 等字段;
5. **测试环境联调**:发起真实清单申报,核对回执(放行/退单),打通三单对碰;
6. **通道切换**:在"系统管理→通道配置"将默认通道由 SIMULATOR 切为 DIRECT(架构已支持切换、双轨灾备);
7. **海关验收**:通过海关的接入测试与验收。

### 2.4 责任分工
| 事项 | 责任方 |
|------|--------|
| 企业备案、电子口岸卡、单一窗口账号、测试环境申请 | 业主/关务 |
| DIRECT 适配器、加签接入、报文字段核对、联调 | 开发方 |
| 接入测试与验收 | 海关 + 业主 + 开发方 |

### 2.5 说明
本项的**软件接口已预留完毕**,剩余是"资质办理 + 真实环境联调",属实施与外部协调工作,非软件单方可"提前完成"。条件到位后,DIRECT 通道对接与联调即可开展。
